Verify an SWS Attention Receipt

Paste any signed SWS attention receipt below. Your browser will verify the Ed25519 (EdDSA) signature against the public key at /.well-known/attention-pubkey.json and render the full seven-layer attestation.
No SWS server is involved in the verification. All math happens in your browser using the W3C WebCrypto API.

Two ways a receipt can be verified

SWS receipts carry two independent integrity claims. (A) Content-binding: every displayed value is included in a deeply key-sorted canonical JSON whose SHA-256 is the receipt hash. Any modification to a displayed value changes the hash and breaks verification. (B) Issuer attestation: when the SWS signing service has stamped the receipt, it carries an Ed25519 (EdDSA) signature over the JWT. Either layer is independently auditable below.

A. Content-bound receipt (SHA-256 over canonical session payload)

Paste the canonical JSON and the claimed hash from a CME or demo session (Firestore demos.<sessionId>.receipt_canonical + receipt_hash). Your browser re-canonicalises and re-hashes locally; if the recomputed hash matches the claimed hash the displayed values are intact.

How this works: The receipt is a compact JWT (header.payload.signature) signed
    with an Ed25519 private key held by SWS. The public key is published at the .well-known
    URL above — anyone, any language, any platform, can verify. We use only your browser's built-in
    WebCrypto API (crypto.subtle.verify with algorithm Ed25519).

    
What each layer proves: (1) environmental gate — real browser, not automation;
    (2) behavioral composite — engagement quality; (3) composition integrity — typing authored vs pasted;
    (4) consent attestation — user actively consented; (5) Ed25519 signing — authentic issuer;
    (6) OpenTimestamps — Bitcoin-anchored, provably not backdated.

    
Ed25519 / EdDSA: RFC 8037 / RFC 8032 / RFC 7515. WebCrypto Ed25519:
    shipped in Chrome 113+, Firefox 130+, Safari 17+.

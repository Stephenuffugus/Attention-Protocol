## 1. DATA FLOW

```
User Interaction → Client-Side Payload Construction → SHA-256 Hashing → localStorage → Firestore Sync (optional)
```

**Critical property:** No raw behavioral data ever leaves the client device. Only aggregate metrics and cryptographic hashes are transmitted.

### Payload Construction
1. User interacts with the application (tap, click, scroll, etc.)
2. Aggregate metrics are collected: duration, count, quality classification
3. A 9-field payload is constructed (event_type, timestamp, session_id, duration_ms, interaction_count, quality_tier, game_id, user_uid, nonce)
4. Payload keys are sorted alphabetically and serialized to JSON
5. SHA-256 hash is computed using SubtleCrypto API
6. Hash + metadata stored to localStorage immediately
7. If authenticated and online, synced to Firestore

### What NEVER enters the payload
- URLs or page identifiers
- Content text or images
- Keystroke content (count only)
- App names or screen identifiers
- User names, emails, or PII
- IP addresses
- Device fingerprints
- Location data (unless opt-in GPS features enabled)

## 2. ENCRYPTION

### In Transit
- **Protocol:** TLS 1.2+
- **Enforced by:** HTTPS for all Firestore and API communications
- **Certificate:** Google-managed SSL for Firebase endpoints

### At Rest
- **localStorage:** Unencrypted on device (standard browser behavior)
- **Firestore:** Google Cloud default encryption (AES-256)
- **Hash payloads:** Irreversible SHA-256 — source data cannot be reconstructed

### Hash Algorithm
- **Algorithm:** SHA-256
- **Implementation:** Browser's SubtleCrypto API (primary), pure JavaScript fallback
- **SubtleCrypto compliance:** FIPS 140-2 validated in Chrome, Firefox, Safari, Edge
- **Collision resistance:** 2^128 operations (computationally infeasible)
- **Output:** 64-character hexadecimal string

## 3. AUTHENTICATION

### Visitor Authentication (Consumer)
- **Method:** Firebase Anonymous Authentication
- **UID generation:** Server-side by Firebase Auth service
- **Session:** Persisted across page reloads, new UID per device/browser
- **Upgrade path:** Anonymous accounts can be linked to email/provider accounts

### Registered User Authentication
- **Methods:** Firebase Auth with Google, email/password, or other providers
- **Token:** Firebase ID token (JWT), auto-refreshed
- **Session:** Persistent until explicit sign-out

### B2B API Authentication
- **Method:** API key + Client ID
- **Key format:** 256-bit randomly generated string
- **Transmission:** `X-SWS-API-Key` header over HTTPS
- **Rotation:** Keys can be rotated without service interruption
- **Scoping:** Each key is scoped to a single client_id

### Webhook Verification
- **Method:** HMAC-SHA256 signature
- **Secret:** Shared secret established during webhook registration
- **Verification:** Recipient computes HMAC-SHA256(secret, raw_body) and compares to `X-SWS-Signature` header

## 4. AUTHORIZATION

### Firestore Security Rules
```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /vaults/{userId}/{document=**} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
  }
}
```

**Enforcement:** Each authenticated user can ONLY read/write their own vault. No cross-user data access is possible. No admin SDK access is exposed to clients.

### API Authorization
- API keys are scoped to a single `client_id`
- Clients can only access sessions generated by their own `client_id`
- No endpoint exposes data from other clients
- Admin operations require separate admin credentials (not available to clients)

## 5. THREAT MODEL

### Threat: Bot/Automation Attacks
- **Risk:** Automated scripts generating fake attention hashes to inflate engagement metrics
- **Mitigation:** 6-signal behavioral analysis composite score
  - Timing entropy (CV): Bots produce CV < 0.25, humans produce CV 0.4-1.5
  - Fitts' Law compliance: Bots fail motor pattern correlation (r < 0.15)
  - Hick's Law compliance: Bots show constant decision time regardless of options
  - Scroll saccade analysis: Bots show zero reading fixations
  - Micro-pause analysis: Bots interact within 0-50ms of content render (humans: 200-2000ms)
  - Touch variance: Bots produce identical touch contact area (variance ≈ 0)
- **Result:** Bots are not blocked but classified to `background` tier (0.25x earning rate), making automation economically pointless

### Threat: Hash Forgery
- **Risk:** Attacker creates fake hashes without genuine attention events
- **Mitigation:** SHA-256 collision resistance (2^128 operations)
- **Additional:** Nonce + timestamp in payload prevents exact hash prediction
- **Result:** Computationally infeasible to forge a hash that matches a legitimate attention event

### Threat: Replay Attacks
- **Risk:** Attacker replays previously generated valid hashes
- **Mitigation:** Each hash includes a unique nonce and timestamp
- **Additional:** Firestore deduplication (same hash cannot be stored twice)
- **Result:** Replayed hashes are detected and rejected

### Threat: Data Tampering
- **Risk:** Attacker modifies a receipt or hash record after generation
- **Mitigation:** Receipt integrity hash — SHA-256 of the receipt's contents
- **Verification:** Any party can recompute the hash and compare to the stored receipt_hash
- **Result:** Tampered receipts are detectable

### Threat: Privacy Breach
- **Risk:** Attacker gains access to stored data and extracts personal information
- **Mitigation:** No PII exists in any payload or storage
- **Data stored:** Duration (ms), interaction count (integer), quality tier (string), SHA-256 hash (hex string), timestamp (ms), anonymous UID
- **Result:** Even with full database access, no personal information can be extracted because none was ever collected

### Threat: Man-in-the-Middle
- **Risk:** Attacker intercepts hash data in transit
- **Mitigation:** TLS 1.2+ for all Firebase/API communications
- **Additional:** Data in transit is the same aggregate metrics + hashes (no PII to intercept)
- **Result:** Standard TLS protection; intercepted data contains no sensitive information

### Threat: localStorage Manipulation
- **Risk:** User manipulates their own hash count in localStorage
- **Mitigation:** Firestore vault is the source of truth for balance; localStorage is a cache
- **Additional:** Hash records include server-verified timestamps and game_id
- **Result:** Client-side manipulation detected on next sync; vault balance authoritative

## 6. INCIDENT RESPONSE

### Monitoring
- Firebase Analytics for anomaly detection on hash generation rates
- Firestore usage monitoring for unusual read/write patterns
- API rate limit monitoring and alerting

### Anomaly Detection
- Sudden spike in hash generation from a single UID → rate limiting triggered
- Timing CV consistently < 0.1 across many sessions → bot classification escalated
- Hash generation without any interaction events → flagged for review

### Response Process
1. Anomaly detected and logged
2. Affected sessions flagged in database
3. If automated attack: rate limit tightened, affected hashes quarantined
4. Client notified if their data quality is impacted
5. Post-incident review and threshold adjustment

## 7. DATA RETENTION

| Data Type | Retention | Deletion Method |
|-----------|----------|-----------------|
| localStorage hashes | Until user clears or 10,000 cap | Automatic trimming + user-initiated delete |
| Firestore hashes | Indefinite (user-controlled) | User-initiated via `SWSPrivacy.deleteAllData()` |
| Firestore balance | Indefinite (user-controlled) | Cascade delete with vault |
| API session data | 90 days | Automatic expiration |
| Webhook logs | 30 days | Automatic expiration |
| API access logs | 90 days | Automatic expiration |

## 8. SUPPLY CHAIN SECURITY

### Dependencies
The core SDK has **zero external dependencies**. It is a single JavaScript file that uses only browser-native APIs:
- `SubtleCrypto` (SHA-256)
- `localStorage` (persistence)
- `TextEncoder` (string encoding)
- `Page Visibility API` (tab detection)
- `Screen Wake Lock API` (ambient mode)
- `Touch Events API` (behavioral analysis)

### Firebase (Optional)
- Firebase JS SDK loaded from Google CDN (`gstatic.com`)
- Firebase services: Auth, Firestore
- Firebase infrastructure: Google Cloud Platform
- SOC 2 Type II compliant (Google Cloud)
